SEC 4.0 Defect Register Structure DFR-STRUCTURE
S1
Critical
System crash, data loss, baseline integrity violation, security breach
SLA: Fix within 24 hrs · CCB emergency convene
S2
Major
Core function broken, wrong output, benchmark failure, rules misfired
SLA: Fix within 72 hrs · CCB notification
S3
Minor
Non-critical functional error, UI cosmetic, edge-case mishandling
SLA: Fix within 2-week sprint · Lead sign-off
S4
Observation
Enhancement opportunity, documentation gap, style deviation
SLA: Backlog — next scheduled release
Defect Identity
  • DEF-ID: DEF-<YYYY>-<SEQ>
  • Date Logged
  • Logged By (Name / Role)
  • Component + Version (from VR)
  • Severity: S1 / S2 / S3 / S4
  • Status: Open / In Progress / Fixed / Verified / Closed
Reproduction & Evidence
  • Short Title (max 80 chars)
  • Detailed Description
  • Steps to Reproduce
  • Expected vs Actual Behaviour
  • Environment (OS / Build / DB instance)
  • Evidence (screenshot / log ref / benchmark run ID)
Baseline Linkage
  • MasterDB Section Reference
  • Baseline Version at Time of Defect
  • CR-ID (if defect triggers change request)
  • Benchmark Job ID (if failed during test)
  • Related Defect IDs (if linked chain)
Resolution
  • Assigned To (Developer)
  • Root Cause Analysis
  • Fix Description
  • Fix Commit / Branch Reference
  • Fixed In Version (from VR)
  • Verified By QA / Date
SLA & Escalation Tracking
  • SLA Due Date (auto from severity)
  • Current SLA Status: On Track / At Risk / Breached
  • Escalation Level: 0 / 1 / 2 / 3
  • Date Escalated (if applicable)
  • Escalated To
Defect States Lifecycle
  • OPEN → assigned on logging
  • IN PROGRESS → developer takes ownership
  • FIXED → patch applied, version incremented
  • IN VERIFICATION → QA testing fix
  • CLOSED → QA verified, baseline re-frozen
  • REJECTED → not reproducible / by design
