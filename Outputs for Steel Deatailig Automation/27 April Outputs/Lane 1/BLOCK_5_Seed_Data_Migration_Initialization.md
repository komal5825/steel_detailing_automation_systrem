## 1. SEED DATA STATUS

### 1.1 Seed Data Architecture

Seed data is the minimal, valid dataset required to initialize a fresh Lane 1 database. It populates:
- Core module registry (organizational structure of validation rules).
- Core field dictionary (metadata for all project-level and object-level fields).
- Core validation rule master (basic rules for L01–L05, L10).
- Source mapping master (field-to-system mappings for TEKLA, REVIT, MANUAL, IFC).
- Source fallback chain (precedence order for multi-source field resolution).

**Seed data scope does NOT include:**
- Project-specific data (created per-project at runtime).
- Validation results (generated by engine at execution time).
- Project stage status (computed during project lifecycle).
- Audit event log (append-only trace; seeded with one SYSTEM init event).

### 1.2 Load Order (Dependency Graph)

```
Dependency Hierarchy:

Level 1 (Root):
├── module_registry
│   └── [Core modules: STRUCTURE, REINFORCEMENT, CONNECTION, GEOMETRY]
│
Level 2 (Field Metadata):
├── field_dictionary
│   └── [FK: module_id → module_registry]
│
Level 3 (Field Properties):
├── source_mapping_master
│   └── [FK: field_id → field_dictionary]
│
├── source_fallback_chain
│   └── [FK: field_id → field_dictionary]
│
Level 4 (Validation Rules):
├── validation_rule_master
│   └── [FK: module_id → module_registry, field_id → field_dictionary]

Level 5 (Project Lifecycle):
├── project_master
│   └── [User creates; no FK dependency on seed data]
│
├── project_stage_status
│   └── [FK: project_id → project_master; created per-stage at runtime]
```

### 1.4 Seed Data Delivery Format

**Option A: SQL Script (Recommended for embedded/desktop deployment)**

Single `.sql` file containing all INSERT statements in dependency order. File: `lane1_seed_data.sql`

```
Advantages:
+ Native SQLite compatibility
+ Fast bulk load via sqlite3 CLI
+ Human-readable and version-controllable
+ Easy to audit and modify

Disadvantages:
- Fragile to syntax errors
- No built-in rollback
```

**Option B: JSON Configuration + Python Loader Script**

Seed data defined in JSON structures; loaded via Python script. Files: `seed_config.json` + `load_seed.py`

```
Example seed_config.json structure:
{
  "modules": [
    {"module_id": "MOD_001", "module_name": "STRUCTURE", ...},
    ...
  ],
  "fields": [
    {"field_id": "FLD_STR_001", "module_id": "MOD_001", ...},
    ...
  ],
  "rules": [
    {"rule_id": "RULE_L01_001", "module_id": "MOD_001", ...},
    ...
  ],
  "mappings": [...],
  "fallbacks": [...]
}

Advantages:
+ Programmatic control; can validate before insert
+ Supports transformations (e.g., UUID generation)
+ Easy rollback via transaction
+ Machine-generated seed data possible

Disadvantages:
- Requires Python runtime
- JSON parsing overhead
- Less transparent than raw SQL
```

**Recommended Hybrid:** Provide both:
1. **SQL script** for quick deployment and manual verification.
2. **Python loader** with transaction safety and error handling for production use.

## 2. MIGRATION / VERSIONING STATUS

### 2.1 Schema Versioning Strategy

**Version Identifier: schema_version**

Located in `project_master.schema_version` (INTEGER, default=1). This tracks which schema version was used to create a project.

**Versioning Scheme: Semantic Versioning**

- MAJOR.MINOR.PATCH (e.g., 1.0.0, 1.1.0, 2.0.0)
- Stored as integer: MAJOR*1000 + MINOR*100 + PATCH (e.g., 1010 = v1.0.1)
- Or: single integer incremented per schema change (simplest: 1, 2, 3, ..., n)

**Recommendation: Linear integer versioning (simplest for SQLite)**

```
Schema versions:
v1 (2026-04-27): Initial lane 1 release (Blocks 1–3, 5)
v2 (TBD): Add advanced layers (L06–L09, L11–L13) — Block 6
v3 (TBD): Add override mechanism, custom plugins
...
```

### 2.2 Migration Architecture

**Layer Architecture:**

```
┌─────────────────────────────────────────────┐
│  Migration Runner (Entry Point)              │
│  - Detects current schema_version            │
│  - Determines upgrade path                   │
│  - Applies incremental migrations            │
│  - Validates integrity post-migration        │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│  Migration Catalog                           │
│  - migrate_v1_to_v2.sql                      │
│  - migrate_v2_to_v3.sql                      │
│  - ... (one per version hop)                 │
│  - down_v2_to_v1.sql (rollback)              │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│  Validation & Verification                   │
│  - PRAGMA integrity_check                    │
│  - FOREIGN KEY check                         │
│  - Index rebuild if needed                   │
└─────────────────────────────────────────────┘
```

### 2.3 Migration Execution Pattern

**For a fresh database:**
1. Check if schema_version table exists.
2. If not, apply full schema (Lane_1__Block1_schema_core.sql).
3. Set schema_version = 1.
4. Load seed data (lane1_seed_data.sql or via Python).
5. Write audit event "SYSTEM: DB initialized, schema_version=1".
6. Ready for use.

**For an existing database:**
1. Query current schema_version.
2. If schema_version < target_version:
   - For each intermediate version (v → v+1):
     - Read migration script (migrate_v{v}_to_v{v+1}.sql).
     - Begin transaction.
     - Execute migration.
     - Validate integrity.
     - Commit.
     - Update schema_version.
3. If any migration fails, rollback and report error.
4. Write audit event "SYSTEM: Schema migrated from vX to vY".

### 2.5 Migration Tool: Python Migration Runner

**File: `migrate.py` (Pseudo-code)**

```python
#!/usr/bin/env python3
"""
Lane 1 SQLite Migration Runner
Manages schema version upgrades with rollback support.
"""

import sqlite3
import os
import sys
import logging
from datetime import datetime

class MigrationRunner:
    def __init__(self, db_path, migrations_dir='migrations'):
        self.db_path = db_path
        self.migrations_dir = migrations_dir
        self.logger = self._setup_logging()
        
    def get_current_version(self):
        """Query current schema_version from database."""
        try:
            conn = sqlite3.connect(self.db_path)
            cursor = conn.cursor()
            cursor.execute("SELECT schema_version FROM project_master LIMIT 1;")
            result = cursor.fetchone()
            conn.close()
            return result[0] if result else None
        except sqlite3.OperationalError:
            # Schema doesn't exist
            return 0
    
    def migrate_to_version(self, target_version, mode='up'):
        """
        Migrate from current version to target version.
        
        Args:
            target_version: Target schema_version (integer)
            mode: 'up' (forward) or 'down' (rollback)
        """
        current = self.get_current_version() or 0
        
        if mode == 'up':
            migrations = list(range(current + 1, target_version + 1))
        elif mode == 'down':
            migrations = list(range(current, target_version, -1))
        else:
            raise ValueError("mode must be 'up' or 'down'")
        
        if not migrations:
            self.logger.info(f"Already at version {current}")
            return True
        
        conn = sqlite3.connect(self.db_path)
        try:
            for version in migrations:
                if mode == 'up':
                    script_name = f'migrate_v{version-1}_to_v{version}.sql'
                    next_version = version
                else:
                    script_name = f'migrate_down_v{version}_to_v{version-1}.sql'
                    next_version = version - 1
                
                script_path = os.path.join(self.migrations_dir, script_name)
                
                if not os.path.exists(script_path):
                    raise FileNotFoundError(f"Migration script not found: {script_path}")
                
                self.logger.info(f"Applying {script_name}...")
                
                with open(script_path, 'r') as f:
                    migration_sql = f.read()
                
                cursor = conn.cursor()
                cursor.executescript(migration_sql)
                
                # Verify integrity
                cursor.execute("PRAGMA integrity_check;")
                result = cursor.fetchone()[0]
                if result != 'ok':
                    raise RuntimeError(f"Integrity check failed: {result}")
                
                # Update schema_version (if not already in script)
                cursor.execute(
                    "INSERT OR REPLACE INTO project_master (project_id, project_name, project_code, schema_version) "
                    "VALUES ('SYS', 'System', 'SYSTEM', ?)",
                    (next_version,)
                )
                
                self.logger.info(f"✓ Migrated to version {next_version}")
            
            conn.commit()
            self.logger.info(f"Migration complete: v{current} → v{target_version}")
            return True
            
        except Exception as e:
            conn.rollback()
            self.logger.error(f"Migration failed: {e}")
            self.logger.error("Transaction rolled back.")
            return False
        finally:
            conn.close()
    
    def initialize_fresh_db(self, target_version=1):
        """Initialize a fresh database with schema and seed data."""
        self.logger.info(f"Initializing fresh database: {self.db_path}")
        
        conn = sqlite3.connect(self.db_path)
        try:
            # Apply initial schema
            schema_script = 'Lane_1__Block1_schema_core.sql'
            with open(schema_script, 'r') as f:
                schema_sql = f.read()
            
            cursor = conn.cursor()
            cursor.executescript(schema_sql)
            
            # Load seed data
            seed_script = 'lane1_seed_data.sql'
            with open(seed_script, 'r') as f:
                seed_sql = f.read()
            
            cursor.executescript(seed_sql)
            
            # Verify
            cursor.execute("PRAGMA integrity_check;")
            result = cursor.fetchone()[0]
            if result != 'ok':
                raise RuntimeError(f"Integrity check failed: {result}")
            
            conn.commit()
            self.logger.info(f"✓ Database initialized successfully (schema_version={target_version})")
            return True
            
        except Exception as e:
            conn.rollback()
            self.logger.error(f"Initialization failed: {e}")
            return False
        finally:
            conn.close()
    
    def _setup_logging(self):
        logging.basicConfig(
```

## 3. INITIALIZATION STATUS

### 3.1 Initialization Sequence

Initialization is the process of preparing a fresh environment (development, testing, or production) with a working Lane 1 database and backend system.

**Sequence Steps:**

```
1. Environment Setup
   ├── Verify Python 3.8+ (or Ruby, if using different stack)
   ├── Verify SQLite 3.22+ (WAL support)
   ├── Verify dependencies (openpyxl for config parsing, if applicable)
   └── Check disk space (estimated 10 MB for DB + indexes)

2. Database Initialization
   ├── Create or locate database file (lane1.db)
   ├── Issue PRAGMA settings (WAL, foreign_keys, cache, synchronous, etc.)
   ├── Create schema from Lane_1__Block1_schema_core.sql
   ├── Verify all tables, constraints, indexes, triggers exist
   └── Log initialization audit event

3. Seed Data Load
   ├── Insert module registry (7 core modules)
   ├── Insert field dictionary (16+ fields)
   ├── Insert validation rule master (15+ rules)
   ├── Insert source mapping master (10+ mappings)
   ├── Insert source fallback chain (8+ chains)
   └── Verify referential integrity (no orphaned FKs)

4. Engine Connectivity Check
   ├── Open connection to initialized DB
   ├── Re-issue PRAGMA foreign_keys = ON per connection
   ├── Load rule_loader from engine_core
   ├── Test rule_loader.load_rules('S1') → should return ≥ 5 rules
   ├── Close connection
   └── Log engine connectivity audit event

5. Verification & Health Check
   ├── Row count checks (projects=0, modules=7, fields=16+, rules=15+)
   ├── Index integrity check (PRAGMA integrity_check)
   ├── Audit log check (should have ≥ 2 entries: init, connectivity)
   ├── Trigger testing (attempt UPDATE on audit_event_log → should ABORT)
   └── Success/failure report

6. Backup & Archive
   ├── Create snapshot backup (backup_lane1_YYYYMMDDTHHmmss.db)
   ├── Generate checksum (SHA256 of backup)
   ├── Archive to backup location
   └── Log backup metadata
```

## 4. BACKUP AND RESTORE MECHANISM

### 4.1 Backup Strategy

**Backup Types:**

| Type | Frequency | Scope | Retention | Use Case |
|------|-----------|-------|-----------|----------|
| **Snapshot Backup** | Before schema migration, before major operations | Full DB file + WAL | 30 days | Point-in-time recovery |
| **Incremental WAL** | Continuous (WAL file) | Only changed pages | 7 days | Quick recovery for recent changes |
| **Archive Backup** | Weekly (or per release) | Full DB | 1 year | Long-term record keeping |

### 4.2 Backup Procedure

**Pre-Backup Checklist:**
1. Ensure no connections to database (or gracefully close active sessions).
2. Issue PRAGMA wal_checkpoint(RESTART) to merge WAL into main DB.
3. Copy DB file + WAL file (if exists).
4. Generate checksum (SHA256).
5. Store in backup location.
6. Log backup metadata.

**Backup Script:**

```bash
#!/bin/bash
# backup_lane1.sh — Backup Lane 1 database

DB_PATH="${1:-lane1.db}"
BACKUP_DIR="${2:-backups}"
TIMESTAMP=$(date -u +%Y%m%dT%H%M%S)
BACKUP_NAME="backup_lane1_${TIMESTAMP}.db"
BACKUP_PATH="${BACKUP_DIR}/${BACKUP_NAME}"

echo "Backing up $DB_PATH to $BACKUP_PATH..."

# Create backup directory
mkdir -p "$BACKUP_DIR"

# Checkpoint WAL (merge into main DB)
sqlite3 "$DB_PATH" "PRAGMA wal_checkpoint(RESTART);" || exit 1

# Copy database file
cp "$DB_PATH" "$BACKUP_PATH" || exit 1

# Generate checksum
CHECKSUM=$(sha256sum "$BACKUP_PATH" | awk '{print $1}')

# Log metadata
cat >> "${BACKUP_DIR}/manifest.json" << EOF
{"timestamp": "$TIMESTAMP", "file": "$BACKUP_NAME", "sha256": "$CHECKSUM"}
EOF

echo "✓ Backup complete: $BACKUP_PATH"
echo "  Checksum: $CHECKSUM"
```

### 4.3 Restore Procedure

**Restore Steps:**
1. Stop all database connections.
2. Locate backup file and verify checksum.
3. Copy backup to target location.
4. Delete associated WAL files (-wal, -shm).
5. Issue PRAGMA integrity_check on restored DB.
6. Re-run initialization (seed data verification).
7. Log restore audit event.

**Restore Script:**

```bash
#!/bin/bash
# restore_lane1.sh — Restore Lane 1 database from backup

BACKUP_PATH="${1:?Backup file path required}"
TARGET_PATH="${2:-lane1.db}"
EXPECTED_SHA256="${3}"

echo "Restoring $BACKUP_PATH to $TARGET_PATH..."

# Verify backup exists
if [[ ! -f "$BACKUP_PATH" ]]; then
    echo "✗ Backup file not found: $BACKUP_PATH"
    exit 1
fi

# Verify checksum if provided
if [[ -n "$EXPECTED_SHA256" ]]; then
    ACTUAL_SHA256=$(sha256sum "$BACKUP_PATH" | awk '{print $1}')
    if [[ "$ACTUAL_SHA256" != "$EXPECTED_SHA256" ]]; then
        echo "✗ Checksum mismatch! Expected: $EXPECTED_SHA256, Got: $ACTUAL_SHA256"
        exit 1
    fi
    echo "✓ Checksum verified: $EXPECTED_SHA256"
fi

# Stop any running connections (application-dependent; example below)
# killall -TERM lane_app 2>/dev/null

# Remove old database and WAL files
rm -f "$TARGET_PATH" "${TARGET_PATH}-wal" "${TARGET_PATH}-shm"

# Copy backup
cp "$BACKUP_PATH" "$TARGET_PATH" || exit 1
echo "✓ Backup copied to $TARGET_PATH"

# Verify integrity
sqlite3 "$TARGET_PATH" "PRAGMA integrity_check;" > /tmp/integrity_check.txt
if grep -q "ok" /tmp/integrity_check.txt; then
    echo "✓ Integrity check passed"
else
    echo "✗ Integrity check failed:"
    cat /tmp/integrity_check.txt
    exit 1
fi

echo "✓ Restore complete"
```
